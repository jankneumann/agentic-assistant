### Observations

- **Main CI has been red since the gen-eval adoption commit.** `pyproject.toml` declares `gen-eval[mcp]` with `[tool.uv.sources]` pointing at `../agentic-coding-tools/packages/gen-eval`, a sibling repo local path. That resolves on the development machine but not in CI, which checks out only this repo. `uv sync` therefore fails during metadata generation before any test runs, producing a 9 second fast fail. Both a878fa3 and 9dfcbf8 show `completed/failure` on main. A `{ path = ... }` source is a development convenience that is fatal in CI and in clean clones.
- **PR #43 carries a second CI bug that is independent of gen-eval.** All 10 failures are one root cause. The new CLI tests in `tests/test_cleanroom_cli.py` and `tests/test_learning_cli.py` wrap invocations in `runner.isolated_filesystem()`, which changes the working directory to a temp dir. `ci.yml` sets `ASSISTANT_PERSONAS_DIR: tests/fixtures/personas` as a relative path, and `src/assistant/core/persona.py` reads it as `Path(env)` without resolving to absolute. Under the temp working directory the relative path resolves nowhere, persona discovery returns empty, the CLI exits 2, and the exit code assertions fail. The suite passes locally only because `tests/conftest.py` uses `os.environ.setdefault` with an absolute path, which is a silent no-op in CI where the workflow has already set the relative value. Verified in both directions in a scratch worktree: relative gives 10 failed / 12 passed, absolute gives 22 passed. Proposed one line fix is `ASSISTANT_PERSONAS_DIR: ${{ github.workspace }}/tests/fixtures/personas`.
- **CI builds the PR head, not the main-merged result.** Run 29698184475 reported `headSha=af255285e`, which is PR #43 head rather than a merge commit. This explains why #43 passed `uv sync` while #41 failed it: #43 branched before gen-eval landed and never touches `pyproject.toml`, so its head has no gen-eval, whereas `refresh-branch` merged main into #41 and pulled gen-eval into that head. Practical consequence: refreshing #43 would import the gen-eval breakage and remove its only clean path to green.
- **The freshly synced merge-pull-requests skill is partially broken in this consuming repo.** The refactored `discover_prs.py` inserts `Path(__file__).resolve().parents[3] / "agent-coordinator"` on `sys.path` and imports `src.github_classifier`. In the source repo the skill lives at `skills/...` where `agent-coordinator` is a sibling, so it resolves. Installed into a consumer under `.claude/skills/...`, `parents[3]` is `.claude`, so the path becomes `.claude/agent-coordinator` and the import fails with ModuleNotFoundError. Triage proceeded with manual classification by branch prefix, which was sufficient because the classification rules are simple string matching. The other three core scripts (`check_staleness`, `analyze_comments`, `merge_pr`) have no coordinator dependency and worked normally. Fix belongs upstream in `agentic-coding-tools/skills/`, followed by a re-run of `install.sh`.
- **Two distinct bugs this session shared one root pattern.** Both the gen-eval `{ path = "../agentic-coding-tools/..." }` dependency and the `discover_prs.py` `parents[3]/agent-coordinator` lookup assume a sibling repo layout that exists only on the development machine. Worth watching as a recurring failure mode whenever this repo reaches toward `agentic-coding-tools`.
